# Девичник Dating App

## Цель

Сделать шуточное браузерное миниприложение в формате dating app parody для девичника. В приложении пользователь просматривает фиксированный набор из 11 анкет, распределяет их по двум категориям (`Нравится` / `Нет`) с помощью свайпов или кнопок, а затем среди понравившихся выбирает одного финального победителя.

Приложение должно быть визуально ярким, нарочито гламурно-розовым, с ощущением "люкс", но при этом оставаться легким, смешным и удобным.

## Основные требования

- Формат: обычный сайт, работающий в браузере.
- Язык интерфейса: русский.
- Целевая природа проекта: шуточный, не production dating service.
- Пользователи и авторизация: не нужны.
- Хранение состояния: не нужно. При обновлении страницы и закрытии вкладки все начинается заново.
- Поддержка устройств: адаптивный интерфейс, mobile-first, но с аккуратной desktop-версией.
- Количество анкет: фиксированное, сейчас 11 штук.
- Категории отбора: только две, `Нравится` и `Нет`.

## Источник данных

Анкеты хранятся локально в виде `.md` файлов. Рядом с ними лежат изображения.

Приложение должно автоматически подхватывать анкеты при сборке проекта.

### Правила структуры данных

- Для каждой анкеты используется один `.md` файл.
- У каждой анкеты одно фото.
- Связь между `.md` и фото идет по одинаковому имени файла.
- Например:
  - `profiles/artyom.md`
  - `profiles/artyom.jpg`

### Формат анкеты

Использовать `frontmatter` плюс основной markdown-текст:

```md
---
name: "Артем"
hobbies: ["спорт", "бар", "путешествия"]
occupation: "продакт"
style: "smart casual"
pets: "собаки"
worst_act: "может пропасть на 3 дня"
---

Люблю активный отдых, вкусно есть и красиво жить.
```

### Поля анкеты

- `name` — имя
- `hobbies` — увлечения
- `occupation` — род деятельности
- `style` — стиль одежды
- `pets` — кошки/собаки
- `worst_act` — самый плохой поступок, на который способен
- `body` — поле "О себе", берется из markdown-body

Если агенту удобнее, можно на этапе сборки преобразовывать markdown в структурированный JSON-массив карточек.

## Пользовательский сценарий

### Экран 1. Старт

Стартовый экран с ярким гламурным визуалом, названием приложения и кнопкой начала.

Что важно:

- Сразу задать тон: розовый, блеск, "люкс", ироничный вайб.
- Не перегружать стартовый экран.
- Можно добавить короткий подзаголовок в духе выбора лучшего кандидата.

### Экран 2. Просмотр анкет

Основной режим приложения: одна карточка анкеты на экране.

Поведение:

- Пользователь видит текущую карточку.
- Может свайпнуть влево/вправо.
- Может нажать кнопку `Нет` или `Нравится`.
- После действия карточка уходит, показывается следующая.

Что должно отображаться на карточке сразу:

- Фото
- Имя
- Увлечения
- Род деятельности
- Стиль одежды
- Кошки/Собаки
- Самый плохой поступок, на который способен
- О себе

Дополнительно:

- Показать прогресс, например `3 из 11`.
- На desktop не полагаться только на жесты: кнопки обязательны.
- Свайпы и кнопки должны делать одно и то же.

### Экран 3. Если никто не понравился

Если после разбора всех карточек список `Нравится` пустой, показать отдельный экран:

- Сообщение в духе "Никто не прошел кастинг"
- Яркий визуал
- Кнопка `Начать заново`

### Экран 4. Финальный выбор

Если есть понравившиеся кандидаты, показать экран только с ними.

Поведение:

- Показать список или сетку карточек понравившихся.
- Пользователь кликает по одному кандидату.
- После выбора открывается финальный экран победителя.

Возврат к предыдущему этапу не обязателен.

### Экран 5. Победитель

Финальный праздничный экран:

- Большая карточка победителя
- Блестящий или сияющий фон
- Эффект праздника на вкус агента
- Кнопка `Начать заново`

Можно использовать:

- конфетти
- сияние
- корону
- анимированные блестки

Важно не уходить в визуальный хаос: эффектно, но читаемо.

## UX-принципы

- Интерфейс должен быть понятен без объяснений.
- Основной сценарий должен проходиться быстро и весело.
- Все ключевые действия должны быть доступны и на телефоне, и на компьютере.
- Текст на карточках должен оставаться читаемым, даже если поля длинные.
- Если текст слишком длинный, нужно аккуратно ограничить переполнение без ломания верстки.
- Анимации должны усиливать настроение, а не тормозить сценарий.

## Визуальное направление

Стиль: гламурно-розовый, нарочито "люкс", с легкой пародией на премиальные dating apps.

Направление:

- насыщенные розовые оттенки
- золотые акценты
- глянцевые поверхности
- мягкие свечения
- крупные карточки
- выразительная типографика
- заметные, но не дешевые анимации

Избегать:

- скучного дефолтного UI
- слишком техничного вида
- обычного "корпоративного" React-шаблона
- перегруженности, из-за которой приложение становится трудно читать

## Технические ожидания

Агенту можно самостоятельно выбрать стек, но рекомендуемый вариант:

- `React + Vite + TypeScript`

Почему:

- быстро поднимается
- удобно работать с локальными данными
- легко сделать адаптивный интерфейс
- просто настроить импорт контента из файлов

Если агент предложит другой стек, он должен кратко объяснить, почему он лучше для этого проекта.

## Архитектурные требования

### Данные

- Данные анкет не должны быть захардкожены в компонентах.
- Нужно сделать отдельный слой загрузки/нормализации анкет из локальных `.md` файлов.
- Нужна валидация или хотя бы понятная обработка ошибок, если файл анкеты заполнен некорректно.

### Настройка текстов

Все пользовательские тексты интерфейса нужно вынести в отдельную конфигурацию или словарь.

Примеры:

- кнопки
- заголовки экранов
- пустые состояния
- подписи прогресса

Цель: чтобы потом можно было быстро поменять тональность или формулировки без переписывания логики.

### Настройка темы

Цвета и базовые визуальные токены должны быть вынесены в отдельные переменные.

Например:

- основные цвета
- акцентные цвета
- фон
- тени
- радиусы
- свечение
- длительности анимаций

Цель: чтобы потом можно было быстро перенастроить стиль без переделки компонентов.

## Что должен сделать агент

1. Инициализировать проект.
2. Настроить структуру папок для приложения, данных анкет и изображений.
3. Реализовать автоматическое чтение `.md` анкет и сопоставление с фото по имени файла.
4. Построить полный сценарий из стартового экрана, свайпа, пустого результата, финального выбора и экрана победителя.
5. Сделать адаптивный интерфейс для mobile и desktop.
6. Добавить выразительный гламурный визуальный стиль.
7. Вынести тексты интерфейса и тему в отдельные конфигурации.
8. Добавить кнопку перезапуска сценария.
9. Подготовить `README` с объяснением, как добавлять и редактировать анкеты.

## Ожидаемая структура проекта

Примерно так:

```text
/src
  /app
  /components
  /screens
  /data
  /theme
  /content
    /profiles
README.md
```

Структура может отличаться, если агент предложит более удачный вариант.

## Обязательные интерфейсные элементы

- стартовый экран
- карточка анкеты
- свайп влево/вправо
- кнопки `Нет` и `Нравится`
- индикатор прогресса
- экран "никто не прошел"
- экран выбора победителя среди понравившихся
- финальный экран победителя
- кнопка `Начать заново`

## Нефункциональные требования

- Приложение должно запускаться локально без сложной настройки.
- Код должен быть достаточно понятным, чтобы потом можно было менять контент.
- Добавление новой анкеты должно быть простой операцией: положить `.md` и фото в нужную папку.
- Интерфейс не должен разваливаться из-за длинных текстов.

## README

Агент должен создать `README.md`, где будет:

- как установить зависимости
- как запустить проект локально
- как собрать production-версию
- куда класть `.md` анкеты
- куда класть фото
- какой формат должен быть у анкеты
- как связываются `.md` и изображение

## Что можно отложить на потом

Не реализовывать сейчас, но проект стоит спроектировать так, чтобы это можно было добавить позже:

- дополнительные эффекты
- больше категорий
- более сложную механику выбора победителя
- сохранение состояния
- админку
- редактирование темы через конфиг

## Критерий успеха

Успешный результат — это локально запускаемое браузерное миниприложение, в котором можно пройти весь шуточный сценарий от просмотра 11 анкет до выбора одного победителя, а контент можно пополнять через markdown-файлы и изображения без правки кода приложения.
